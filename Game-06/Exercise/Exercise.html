Guia tecnica - No es la solucion, es el mapa para encontrarla

Puzzle de fragmentos: "Donde esta la llave?"

Este documento explica como esta armado el proyecto que descomprimiste, como se representan las piezas (Tile) y sus coordenadas, como se relaciona todo con la imagen fuente, y en que metodo exacto tienes que escribir tu algoritmo.

01El problema, en una frase

Una imagen de 150x150 px se corto en 4 fragmentos de 75x75 (una grilla 2x2). Cada fragmento se guarda en un Tile que sabe permanentemente que trozo de la imagen le toca mostrar. Lo que tu codigo debe hacer es reordenar esa lista de Tiles (y aplicarles la rotacion correcta) para que cada uno termine en la casilla que le corresponde segun una solucion fija - al lograrlo, se revela una silueta oculta distinta del recorte original.

No es un juego de "ordenar por numero". El fragmento que salio de la esquina superior izquierda de la imagen no necesariamente termina en la esquina superior izquierda del tablero resuelto. A donde debe ir cada fragmento lo dice un diccionario llamado SOLUTION, no el orden en que se corto la imagen.

03Coordenadas: grilla (i, j) vs pixeles (x, y)

Cada Tile vive en dos sistemas de coordenadas a la vez, y hay que no mezclarlos:

  • i, j - fila y columna logica dentro del tablero (0 o 1 en cada eje, porque el tablero es 2x2).
  • x, y - posicion en pixeles dentro del tablero, usada solo para dibujar.

La conversion es directa porque TILE_SIZE = 75:

# src/Tile.py - al crear el Tile
self.i = self.y // settings.TILE_SIZE   # pixel -> fila
self.j = self.x // settings.TILE_SIZE   # pixel -> columna

Al dibujar, se suma un desplazamiento fijo (BOARD_OFFSET_X/Y) que solo sirve para centrar el tablero 2x2 dentro de la pantalla virtual - no afecta la logica de i, j, solo el punto donde se pinta cada tile:

BOARD_OFFSET_X = VIRTUAL_WIDTH / 2  - (TILE_SIZE*2)/2
BOARD_OFFSET_Y = VIRTUAL_HEIGHT / 2 - (TILE_SIZE*2)/2

04Anatomia de un Tile (el de la raiz - el que importa para resolver)

Este es el Tile "completo" que usa SortingPuzzle. Cada atributo responde una pregunta distinta:

AtributoResponde...
id / frameQue fragmento de imagen soy? (fijo para siempre, nunca cambia)
i, jEn que casilla estoy ahora mismo?
rotationCuantos giros de 90 grados llevo acumulados ahora mismo? (0-3)
target_i, target_jEn que casilla debo terminar?
target_rotationCon que rotacion debo terminar?
target_indexAtajo: target_i * BOARD_WIDTH + target_j - la posicion destino como un solo numero, util para ordenar una lista.

is_correct_position() compara i == target_i, j == target_j y rotation == target_rotation. El puzzle completo se resuelve cuando esto es cierto para los 4 tiles a la vez (eso es exactamente lo que revisa SortingPuzzle.is_sorted()).

05Como funciona la imagen

La textura fuente es assets/textures/imagen.jpg, de 150x150 px. Con TILE_SIZE = 75, generate_frames la corta en una grilla 2x2 de recuadros de 75x75, numerados por fila (row-major): fila 0 primero (columnas 0,1), luego fila 1 (columnas 0,1).

Ese numero de recuadro es el frame, y en SortingPuzzle.new_tiles() se usa tambien como id - o sea, el id de un Tile es su posicion de origen en la imagen, calculada asi:

origin_i, origin_j = divmod(value, settings.BOARD_WIDTH)  # value = 0,1,2,3

Estos son los 4 fragmentos reales de tu imagen, ampliados, junto a su id/frame y su casilla de origen (fila, columna):

id / frame = 0origen (i=0, j=0)
id / frame = 1origen (i=0, j=1)
id / frame = 2origen (i=1, j=0)
id / frame = 3origen (i=1, j=1)

Ordenar el puzzle nunca cambia id ni frame de un Tile - un fragmento siempre muestra el mismo trozo de imagen. Lo unico que cambia al "resolver" es la casilla donde ese fragmento vive y cuantas veces esta rotado.

06La solucion: de donde viene cada fragmento y a donde debe ir

SOLUTION es un diccionario fijo, indexado por la posicion de origen (la casilla de la que se recorto el fragmento en la imagen), que da la casilla destino y la rotacion destino:

SOLUTION = {
    (0, 0): (0, 1, 0),   # origen fila0,col0 -> destino fila0,col1, sin rotar
    (0, 1): (1, 0, 0),   # origen fila0,col1 -> destino fila1,col0, sin rotar
    (1, 0): (0, 0, 2),   # origen fila1,col0 -> destino fila0,col0, 2 giros (180 grados)
    (1, 1): (1, 1, 2),   # origen fila1,col1 -> destino fila1,col1, 2 giros (180 grados)
}

Nota que dos fragmentos si quedan en su casilla de origen (los de la fila 1), pero necesitan 2 giros (180 grados) para calzar; y los dos de la fila 0 se intercambian de casilla entre si sin rotar. Esta mezcla es justamente lo que arma la silueta oculta en vez de reconstruir la imagen original intacta.

07Rotacion: que significa cada valor

rotation cuenta giros de 90 grados en sentido horario, y va de 0 a 3. target_rotation usa la misma escala:

# Tile.rotate() en el demo jugable, para referencia de la escala:
self.rotation = (self.rotation + 90) % 360   # ahi se cuenta en grados (0/90/180/270)

# en cambio Tile.py (raiz) y target_rotation cuentan "pasos" 0..3:
image = pygame.transform.rotate(image, -90 * self.rotation)

Cuidado con esta diferencia si comparas ambos archivos: el Tile del demo jugable guarda grados (0/90/180/270); el Tile de la raiz (el del ejercicio) guarda pasos (0/1/2/3). target_rotation = 2 en SOLUTION equivale a 180 grados.

08Donde escribir tu codigoesto es lo unico que debes tocar

El unico metodo que debes implementar es SortingPuzzle.sort() en SortingPuzzle.py. Ahora mismo su cuerpo es pass, con una implementacion de referencia (Bubble Sort) comentada arriba como pista de formato - no de que debas usar exactamente ese algoritmo.

Contrato de la funcion: recibe self.tiles (una lista de 4 Tiles, en cualquier orden, cada uno con su target_i/target_j/target_rotation ya asignado por new_tiles()). Al terminar, self.tiles debe quedar reordenada en su lugar segun la casilla destino de cada Tile (usa target_index para comparar), y cada Tile debe tener rotation igual a su target_rotation.

Como saber si funciono, sin necesitar el motor grafico completo:

from SortingPuzzle import SortingPuzzle

puzzle = SortingPuzzle(SortingPuzzle.new_tiles())
puzzle.sort()
print(puzzle.is_sorted())  # debe imprimir True cuando tu algoritmo este completo
  • Antes de ordenar, cada Tile ya trae su target_index calculado - no necesitas leer SOLUTION directamente dentro de sort().
  • Ordenar la lista no alcanza: tambien hay que copiar target_rotation a rotation en cada Tile.
  • Cualquier algoritmo de ordenamiento sirve (burbuja, seleccion, insercion, sorted() con key=); con solo 4 elementos el rendimiento no es el punto del ejercicio.
  • is_sorted() ya esta implementado - usalo para verificar, no hace falta tocarlo.

09Resumen rapido

  • El tablero es 2x2, cada casilla de 75x75 px, imagen fuente de 150x150 px.
  • id/frame = fragmento de imagen, fijo. i, j = casilla actual, cambia al ordenar. target_i, target_j, target_rotation = casilla y giro que exige SOLUTION.
  • El mapeo origen-destino vive en SOLUTION, indexado por casilla de origen en la imagen, no por id de tablero actual.
  • Tu codigo va en SortingPuzzle.sort(): reordena self.tiles por target_index y aplica target_rotation a rotation.
  • Verifica con is_sorted() antes de conectarlo al motor grafico completo.
